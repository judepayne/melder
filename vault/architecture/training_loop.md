# Synthetic Fine-Tuning Loop

Measures how much embedding model accuracy could improve through iterative fine-tuning, using only synthetic data generated by `benchmarks/data/generate.py`.

## Location

`benchmarks/accuracy/training/` — all scripts, config, and committed outputs.

## Entry Point

Run from project root:
    python benchmarks/accuracy/training/run.py --rounds 5

## Terminology

- **Round**: one full cycle of generate → match → evaluate → extract pairs → fine-tune → export. Use "rounds", not "epochs".
- **Epoch**: one pass through the training pairs within a single fine-tuning call (3 epochs per round by default).

## Loop Structure

```
Round 0 (baseline):
  generate(A₀, B₀, seed=0) → run melder (base model) → evaluate → extract pairs

Round N:
  fine-tune(base model + all pairs 0..N-1) → export ONNX
  generate(Aₙ, Bₙ, seed=N) → run melder (modelₙ) → evaluate → extract pairs

Holdout (seed=9999, never trained on):
  evaluated at every round for a clean learning curve
```

## Key Design Decisions

- **Full retrain from base each round** — not incremental. All accumulated pairs are used each time, starting from `all-MiniLM-L6-v2`. Avoids catastrophic forgetting.
- **Fresh data each round** — different seed per round forces the model to generalise, not memorise.
- **Continuous labels** — 1.0 (matched), 0.7 (ambiguous), 0.0 (hard negative). Uses `CosineSimilarityLoss`.
- **Ground truth from generator** — `_true_a_id` and `_match_type` fields on B records enable perfect evaluation without human annotation.

## Scripts

| File | Role |
|---|---|
| `run.py` | Orchestration — only entry point |
| `evaluate.py` | Ground-truth classification using `_true_a_id` |
| `pairs.py` | Extracts `(a_text, b_text, label)` training pairs |
| `finetune.py` | sentence-transformers training (MPS/CUDA/CPU auto-detect) |
| `export.py` | ONNX export via `optimum-cli` |
| `plot.py` | Learning curve chart |
| `config.yaml` | Base melder config template (`top_n: 5`, `ann_candidates: 50`) |
| `requirements.txt` | `optimum[onnxruntime]`, `datasets`, `accelerate`, `pyyaml`, `matplotlib`, `pandas` |

## Output

- `rounds/round_N/metrics.json` — per-round accuracy stats (committed)
- `rounds/round_N/pairs.csv` — training pairs used (committed)
- `results/metrics.csv` — accumulated learning curve table (committed)
- `results/learning_curve.png` — 2×2 chart (committed)
- `models/round_N/model.onnx` — fine-tuned ONNX (gitignored, ~86 MB each)

## Timing (M3 MacBook Air, 10k × 10k)

- Per round: ~17s meld (cold) + ~2.1 min fine-tune (grows with rounds) + ~6s ONNX export
- 5 rounds total: ~25 minutes
- 10 rounds total: ~1.5 hours

## First Run Results (5 rounds, seeds 0–4, holdout seed 9999)

### Results table

| Round | Train Prec | Train Recall | Train FP | Holdout Prec | Holdout Recall | Holdout FP |
|---|---|---|---|---|---|---|
| 0 (base) | 93.3% | 92.8% | 441 | 93.2% | 92.2% | 451 |
| 1 | 79.6% | 95.4% | 1,641 | 32.0% | 7.0% | 989 |
| 2 | 89.8% | 99.5% | 752 | 0.7% | 0.0% | 135 |
| 3 | 90.1% | 99.9% | 732 | 0.0% | 0.0% | 3 |
| 4 | 89.8% | 99.8% | 758 | 0.0% | 0.0% | 0 |

### What happened

Training performance improved substantially after round 0: recall climbed to ~99%, FPs stabilised around 750. But the holdout collapsed entirely — by round 3 the fine-tuned model makes zero matches on the holdout dataset.

### Root cause: holdout design flaw

The experiment used different Faker seeds for training (0–4) and holdout (9999). Different seeds produce completely different company names with no token overlap. The model was not learning generalizable noise-handling rules — it was memorising the specific entity names from seeds 0–4. When presented with seed 9999 names it had never seen, cosine similarity fell below the 0.85 auto_match threshold and nothing matched.

The 99% training recall is the giveaway: that is memorisation, not learning.

### What the experiment actually tested vs what it should test

- **Tested:** "Can the model recognise entity names from seeds 0–4?" — Yes, trivially.
- **Should test:** "Has the model learned noise-invariant representations of entity names?" — Unknown, because the holdout used different entity names.

### The fix

The holdout should share the same A-side entity pool as the training data, with different noise draws on the B-side. This tests noise generalisation (what we care about) rather than entity name memorisation (what we don't).

For real-world fine-tuning this problem does not arise: training and production data share the same counterparty universe, so the model learns to distinguish the actual confusable entities in the deployment data.

### Conclusion

Synthetic fine-tuning with independent random seeds is not a valid experimental design for measuring generalisation. The base model (round 0) remains the best performing configuration: 93.3% precision, 92.8% recall, 451 FPs — and it generalises correctly to unseen data.
