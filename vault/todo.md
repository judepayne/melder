- [ ] **Investigate silent GPU encode deadlock with pool_size > 1** — On machines with smaller GPUs (e.g. M3 Air), `encoder_pool_size > 1` with `encoder_device: gpu` (CoreML) causes `encode_and_upsert` to silently hang during batch encoding. The dedicated rayon thread pool (sized to pool_size) creates multiple concurrent CoreML ONNX sessions that contend on the GPU. With pool_size=12 the process hangs immediately (no progress logged). With pool_size=4 it runs but decelerates (~49 rec/s vs ~170 rec/s with pool_size=1). Likely cause: CoreML resource exhaustion when multiple sessions compete for GPU access. Agreed design direction: keep multi-session GPU mode available, but add a CoreML-specific watchdog in `encode_and_upsert()` that tracks completed chunks/progress; if no chunk completes within a fixed timeout, log a loud fatal message explaining the known CoreML stall and recommending `encoder_pool_size: 1` or `encoder_device: cpu`, then abort the run with a nonzero exit rather than attempting in-process fallback. Also emit an upfront warning at encoder init on macOS GPU when `encoder_pool_size > 1`. Discovered 2026-04-09 on M3 MacBook Air; the M1 Ultra handled pool_size=12 without issue.
